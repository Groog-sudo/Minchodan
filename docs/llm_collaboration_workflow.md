# LLM 협업 작업 분담 가이드

> **작성일**: 2026-06-30
> **버전**: v0.1.0
> **적용 대상**: `th` 브랜치 YOLO/TTS MVP 및 이후 개인 담당 구현

---

## 1. 목적

이 프로젝트는 결과물 제출만이 아니라 발표와 학습이 함께 필요한 최종 프로젝트입니다. 따라서 LLM 에이전트는 코드를 전부 대신 작성하지 않고, 담당자가 직접 이해하고 설명해야 하는 핵심 로직을 남겨 두어야 합니다.

핵심 원칙은 다음과 같습니다.

| 구분 | 원칙 |
| --- | --- |
| **담당자 역할** | 발표에서 설명해야 하는 판단 로직과 실행 흐름을 직접 작성합니다. |
| **LLM 역할** | 프로젝트 연결, 반복 코드, 오류 처리, 테스트, 문서화를 보조합니다. |
| **작업 방식** | 큰 구조는 LLM이 제안하고, 핵심 함수는 담당자가 직접 입력한 뒤 LLM이 검증합니다. |
| **완료 기준** | 담당자가 코드의 입력, 처리, 출력, 검증 결과를 자기 말로 설명할 수 있어야 합니다. |

## 2. 작업 분담 기준

| 영역 | 담당자 직접 작성 | LLM 보조 작성 |
| --- | --- | --- |
| **핵심 알고리즘** | bbox 방향 계산, 거리 계산, 위험도 규칙, 안내 문장 규칙 | 타입 힌트 정리, 예외 처리, 기존 모듈 연결 |
| **데모 실행** | PowerShell 명령어 실행, 결과 해석, 발표용 로그 확인 | argparse, 경로 처리, JSON 저장, 출력 포맷 정리 |
| **YOLO/TTS 연결** | `direction`, `risk_level`, `message_hint.text` 생성 규칙 | YOLO detector 호출부, 결과 직렬화, React Native 서비스 연결 |
| **테스트** | 직접 명령 실행, 성공/실패 로그 확인 | pytest 보강, 실패 원인 분석, 캐시/환경 경고 분리 |
| **문서화** | 본인이 설명할 검증 문장과 발표 흐름 확인 | changelog, 검증 보고서, 문서 인덱스 갱신 |

## 3. 담당자가 직접 치는 우선 영역

`th` 브랜치 YOLO/TTS MVP에서는 다음 코드를 담당자가 직접 작성하거나, LLM이 만든 코드를 담당자가 다시 읽고 수정해야 합니다.

| 우선순위 | 파일 | 직접 이해해야 하는 부분 | 발표 설명 포인트 |
| --- | --- | --- | --- |
| 1 | `server/detection/direction.py` | `estimate_direction`, `estimate_distance` | bbox 중심점과 면적 비율로 방향과 거리를 계산합니다. |
| 2 | `server/detection/risk_rules.py` | `estimate_risk_level`, `build_message_text` | 객체 종류, 방향, 거리로 위험도와 안내 문장을 결정합니다. |
| 3 | `scripts/run_yolo_tts_demo.py` | 입력 이미지, YOLO 결과, JSON 출력 흐름 | 이미지 입력부터 TTS 문장까지의 데모 파이프라인을 설명합니다. |
| 4 | `client/src/services/ttsPlayer.ts` | `speak(hint)`의 REFLEX 선점 처리 | 서버가 만든 `message_hint.text`를 단말 TTS가 읽습니다. |
| 5 | PowerShell 명령어 | pytest, 데모 실행, 결과 파일 확인 | 검증을 직접 수행했다는 증거를 제시합니다. |

## 4. LLM이 맡아야 하는 영역

LLM은 담당자의 학습 시간을 빼앗지 않도록 다음 영역을 중심으로 맡습니다.

| 영역 | LLM 작업 |
| --- | --- |
| **프로젝트 구조 파악** | 기존 파일, 스키마, 테스트, 브랜치 규칙을 읽고 충돌 가능성을 줄입니다. |
| **반복 코드 작성** | CLI 인자, 파일 탐색, JSON 저장, 리포트 생성 같은 반복 작업을 작성합니다. |
| **연결부 구현** | 기존 YOLO detector, message builder, React Native TTS 서비스를 연결합니다. |
| **오류 처리** | Windows 경로, import path, venv 의존성, pytest 경고를 분석합니다. |
| **검증 자동화** | `py_compile`, `pytest`, 샘플 실행 명령을 정리하고 결과를 기록합니다. |
| **문서 반영** | `docs/changelogs/th.md`, 검증 문서, 실행 절차를 갱신합니다. |

## 5. 권장 협업 절차

| 순서 | 담당 | 작업 |
| --- | --- | --- |
| 1 | LLM | 이번 작업의 큰 흐름과 수정 대상 파일을 3~5줄로 설명합니다. |
| 2 | LLM | 담당자가 직접 칠 코드 범위를 20~40줄 이하로 잘라 제시합니다. |
| 3 | 담당자 | 해당 코드 또는 명령어를 직접 입력하고 실행합니다. |
| 4 | 담당자 | 오류 로그나 실행 결과를 그대로 공유합니다. |
| 5 | LLM | 오류 원인을 설명하고 최소 수정안을 제시합니다. |
| 6 | LLM | 반복 연결 코드, 문서, 테스트 보강을 처리합니다. |
| 7 | 담당자 | 최종 실행 명령과 결과 문장을 직접 확인합니다. |

## 6. 코드 작성 단위 기준

| 코드 크기 | 처리 방식 |
| --- | --- |
| **10줄 이하** | 담당자가 직접 치는 것을 기본으로 합니다. |
| **20~40줄** | LLM이 큰 틀을 주고 담당자가 핵심 조건문과 반환값을 직접 작성합니다. |
| **40줄 초과** | LLM이 스캐폴딩을 작성하되, 핵심 함수 1~2개는 담당자 입력 영역으로 분리합니다. |
| **반복/보일러플레이트** | LLM이 작성합니다. 담당자는 동작 흐름만 확인합니다. |
| **발표 핵심 로직** | LLM이 대신 완성하지 않고 담당자에게 설명과 입력 기회를 먼저 제공합니다. |

## 7. YOLO/TTS MVP 적용 예시

| 단계 | 담당자 직접 설명 | LLM 보조 |
| --- | --- | --- |
| **데이터 확인** | 원천데이터는 `.env` 경로로 참조하고 Git에 올리지 않는 이유 | 데이터셋 스캔 스크립트 작성 |
| **샘플 선정** | 바운딩박스 XML이 CVAT 구조이고 JPG와 함께 있다는 점 | XML 파싱 리포트 작성 |
| **YOLO 추론** | 이미지 입력 후 객체명, confidence, bbox가 나온다는 점 | YOLO 호출 스크립트 작성 |
| **방향 계산** | bbox 중심 x좌표를 화면 비율로 나누는 방식 | 타입 힌트, 가드레일 작성 |
| **위험도 계산** | 객체 종류와 거리로 high/medium/low를 정하는 규칙 | JSON 결과 병합 |
| **TTS 연결** | `message_hint.text`를 단말 TTS가 읽는 흐름 | React Native 서비스 파일 정리 |

## 8. 금지되는 작업 방식

| 금지 항목 | 이유 |
| --- | --- |
| **LLM이 전체 기능을 무단 완성** | 담당자가 발표 때 코드 흐름을 설명하기 어렵습니다. |
| **담당자 확인 없이 핵심 규칙 변경** | 위험도, 방향, 문장 생성은 발표 핵심 로직입니다. |
| **검증 명령을 LLM만 실행하고 끝냄** | 담당자가 실행 증거와 결과 해석을 가져가기 어렵습니다. |
| **원천데이터 전체 복사** | Git 추적 위험과 디스크 부담이 큽니다. 샘플만 사용합니다. |
| **반사 경로에 LLM/RAG/실시간 TTS 연결** | 프로젝트의 이중 경로 원칙을 위반합니다. |

## 9. 발표용 설명 템플릿

담당자는 구현 후 다음 흐름을 자기 말로 설명할 수 있어야 합니다.

```txt
이미지를 YOLO에 넣으면 객체명, confidence, bbox가 나옵니다.
bbox 중심 x좌표로 왼쪽, 정면, 오른쪽 방향을 계산합니다.
bbox 면적 비율로 가까움 정도를 near, medium, far로 나눕니다.
객체 종류와 거리 정보를 합쳐 위험도를 결정합니다.
마지막으로 message_hint = {id, type, text}를 만들어 단말 TTS가 읽게 합니다.
```

## 10. 다른 LLM 에이전트에게 주는 지시

이 저장소에서 `th` 브랜치 또는 YOLO/TTS MVP를 다룰 때는 다음을 우선 적용합니다.

| 지시 | 내용 |
| --- | --- |
| **먼저 확인** | `AGENTS.md`, `SKILLS.md`, 본 문서를 읽고 담당자 직접 작성 영역을 확인합니다. |
| **먼저 묻기** | 핵심 판단 로직을 새로 작성하기 전에 담당자가 직접 칠지 확인합니다. |
| **작게 나누기** | 20~40줄 단위로 코드를 나눠 설명하고, 담당자가 실행하게 합니다. |
| **검증 공유** | 실행 결과는 담당자가 발표에 쓸 수 있는 문장으로 정리합니다. |
| **문서 갱신** | 주요 작업 후 `docs/changelogs/th.md`에 변경과 검증 결과를 남깁니다. |
