# Minchodan RAG 백엔드(4·5단계) 폴더 및 파일 구조 가이드북

> **작성일**: 2026-06-26
> **버전**: v1.0.0
> **설계 기준**: `docs/minchodan_design_note.md` 4·5단계
> **코딩 패턴 기준**: `docs/course_codebase_guide.md`

본 문서는 Minchodan 프로젝트의 **4단계 (RAG 지식 DB 구축)** 및 **5단계 (실시간 검색 및 Fallback 안전망)** 백엔드 모듈에 참여하는 폴더와 파일들의 의미, 구조, 역할을 입문 개발자의 눈높이에 맞추어 상세히 풀어 설명합니다.

## 1. 4·5단계 핵심 프로세스 이해하기

입문 개발자가 전체 소스코드의 구조를 파악하기 전에, 시스템의 핵심 흐름과 기술적 역할을 먼저 명확히 파악할 수 있도록 작동 원리를 요약합니다.

### 1.1 4단계: RAG 지식 DB 구축 (오프라인 파이프라인)
4단계는 로컬 환경에 보조 지식 데이터를 수집 및 가공하여 검색용 데이터베이스를 조립해 두는 오프라인 준비 단계입니다. (이 과정은 기존 인공지능 모델 내부의 신경망 가중치를 직접 변경하는 파인튜닝/학습 단계가 아니라, 외부 참조용 지식 기지를 쌓는 과정입니다.)

1. **프레임 추출**: 긴 보행 가이드 동영상 파일(mp4 등)을 1초에 1프레임(1fps) 간격으로 쪼개어 정적 이미지 파일들로 자동 분할합니다.
2. **pHash (지각 해시) 중복 제거**: 이미지 안의 픽셀 밝기 변화 및 시각적 구도를 분석하여 pHash 해시값으로 변환합니다. 시각적으로 차이가 없는 거의 동일한 프레임(예: 보행자가 일시 정지하여 정지 화면이 반복되는 구간)을 수학적으로 비교하여 걸러냅니다. (이를 통해 불필요한 이미지 저장 공간 낭비와 불필요한 VLM API 호출 요금 발생을 원천 차단합니다.)
3. **VLM 캡셔닝 및 ChromaDB 생성**: 중복이 차단된 고유 프레임에 대해 구글의 Gemini VLM(시각 언어 모델) API를 호출하여 한글로 묘사된 상세 설명 텍스트(캡션)를 추출합니다. 이후 이 설명 텍스트들을 텍스트 조각으로 변환하고 의미 임베딩을 거쳐 로컬 벡터 DB인 ChromaDB 인덱스로 하드디스크에 영구 저장합니다.

### 1.2 5단계: 실시간 검색 및 Fallback 안전망 (온라인 추론)
5단계는 보행 과정에서 실시간으로 사물을 회피할 수 있는 행동 지침을 빠르게 찾아내어 TTS 채널로 넘기는 서비스 제공 단계입니다.

1. **임베딩 및 의미 검색**: 카메라를 통해 실시간으로 3단계 YOLO 탐지 결과(예: `kickboard`, `bollard` 등 위험물 라벨)를 받으면, 이 라벨 정보를 질의문으로 삼아 실시간 의미 검색을 실행합니다. 4단계에서 생성해둔 ChromaDB에 질문을 던져 가장 연관성이 높은 장애물 대처 수칙 템플릿 문서를 순식간에 검색해 냅니다.
2. **Fallback (안전망)**: 만약 야외 보행 중 네트워크가 단절되어 로컬 DB에 에러가 발생하거나, ChromaDB 검색 결과에 질문한 사물이 존재하지 않는 경우(미적중) 시스템 작동이 멈추면 시각장애인 보행자에게 치명적인 안전 위험이 발생합니다. 이를 원천 차단하기 위해 예외 차단 가드레일을 설치하고, 검색 실패 시 사전에 등록된 룰 기반 안전 대처 수칙(Fallback)이 중단 없이 즉시 TTS 채널로 송출되도록 우회 경로를 안전하게 구축하였습니다.

## 2. 디렉토리 및 파일 역할 정의서

전체 모듈 파일들은 지침에 따라 `server/rag/` 패키지 구조 하위에 규칙대로 분할 배치되어 있습니다.

| 경로 및 파일명 | 의미 및 기술적 역할 |
| :--- | :--- |
| **server/rag/** | 4·5단계 RAG 관련 추상화 팩토리 및 검색/안전망 모듈을 모아둔 전용 패키지 루트 폴더입니다. |
| ├── **build/** | 4단계 오프라인 DB 구축을 위한 소스코드 스크립트를 격리 보관한 디렉토리입니다. |
| │   ├── **frame_extractor.py** | 비디오 파일을 1초 단위로 분할하여 개별 이미지 파일로 저장하며, 버퍼 오류 예외를 가드합니다. |
| │   ├── **dedup_phash.py** | 해밍 거리(Hamming Distance) 임계치를 이용해 이미지 내용의 고유성을 식별하고 유사 중복 이미지를 자동 필터링합니다. |
| │   ├── **gemini_captioner.py** | 구글의 Gemini VLM API를 이용해 이미지의 시각 정보를 한글 텍스트 설명문으로 자동 복원합니다. |
| │   └── **db_builder.py** | 위 3개의 모듈을 차례대로 파이프라인 형태로 가동하여 최종 ChromaDB를 생성해 주는 오케스트레이터입니다. |
| ├── **shared/** | 4단계와 5단계 전반에서 공유되어 사용되는 데이터 형식을 격리한 폴더입니다. |
| │   └── **labels.py** | 3단계 YOLO 모델이 탐지한 라벨명과 RAG가 매칭할 지식 기지의 라벨명이 1:1로 정확하게 일치하도록 통제하는 단일 진실 공급원(SSOT) 상수 정의서입니다. |
| ├── **embedding_engine_factory.py** | Ollama nomic-embed-text 로컬 호출 및 오프라인 환경을 위한 결정론적 Mock 임베딩 인스턴스를 공급하는 팩토리 클래스입니다. |
| ├── **vector_db_factory.py** | 영구 저장소에 쓸 ChromaDB 객체를 생성할 때 임베딩 모델을 동적 주입받아 바인딩하고 파일 경로 유효성을 검사하는 팩토리 클래스입니다. |
| ├── **retriever.py** | 주입된 ChromaDB 객체를 대상으로 YOLO 탐지 라벨을 던져 코사인 유사도 방식으로 수칙 지침을 검색해 오고, 불일치 라벨을 차단하는 검색 모듈입니다. |
| └── **fallback.py** | RAG 검색 도중 예외가 나거나 적당한 검색 결과가 존재하지 않을 때, 상호 충돌 없이 즉각 룰 기반의 기본 대처 수칙을 반환해 주는 안전 우회 모듈입니다. |

## 3. 단위 및 E2E 테스트 구조의 역할

작성된 모든 소스코드가 안전하고 정합성 있게 동작하는지 증명하기 위해 `tests/` 폴더 내부에 단위 테스트 코드들을 배치하여 리팩토링 검증을 자동화하고 있습니다.

* **개별 단위 테스트 (Unit Test)**:
  각 파일의 `if __name__ == "__main__":` 블록 및 `tests/test_*.py` 파일들을 활용해 DB 팩토리 생성, 중복 제거 해시 비교, 룰 딕셔너리 안전 접근 등의 개별 모듈 로직만을 신속히 검사합니다.
* **통합 E2E 테스트 (E2E Test)**:
  `tests/test_e2e_pipeline.py`를 활용해 "비디오 생성 -> 4단계 DB 빌드 -> 5단계 팩토리 로드 -> Retriever 검색 실행 -> 검색 결과가 없을 때 Fallback 안전망 작동"에 이르는 전체 RAG 시나리오의 결합과 예외 복구 능력을 한꺼번에 완벽히 검증합니다.
